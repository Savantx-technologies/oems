# Role Flow

## 1. SuperAdmin Flow

### Authentication Flow

Superadmin routes live under `/superadmin`.

Guest flow:

- login form
- password login
- OTP send
- OTP verify

Authenticated flow:

- dashboard access
- logout
- profile and password update

### Business Flow

Typical superadmin flow:

1. Login to the superadmin panel.
2. Create or update a school.
3. Create or update a school admin.
4. Manage admin accounts across schools.
5. Approve staff requests generated by school admins.
6. Review admin requests such as block/unblock staff.
7. Monitor exams across schools.
8. Configure platform rules and role permissions.
9. Review reports, logs, and analytics.

### Main SuperAdmin Modules

- Dashboard
- Security Logs
- School Management
- Admin Management
- Sub Super Admin Management
- Staff Request Approval
- Admin Request Handling
- Student Management
- Exam Oversight
- Live Monitoring
- Notifications
- Roles & Permissions
- Reports
- System Settings

### Permission Model

Superadmin currently supports:

- `superadmin`
- `sub_superadmin`

`superadmin` has full access.

`sub_superadmin` access is controlled via sidebar/section permissions stored in role defaults plus optional custom `permissions`.

## 2. Admin Flow

### Authentication Flow

Admin routes live under `/admin`.

Authentication supports:

- login
- OTP flow
- mobile OTP flow
- logout
- password update with OTP support

### Admin as a Product Actor

At the product level, `admin` is one actor type.

Inside that actor type, current sub-roles are:

- `school_admin`
- `sub_admin`
- `invigilator`
- `staff`

### Typical School Admin Flow

1. Login to the school admin panel.
2. Configure school-level settings if needed.
3. Create or request staff members.
4. Add students individually or via bulk upload.
5. Build question bank and passages.
6. Create exam.
7. Attach questions to the exam.
8. Schedule the exam.
9. Publish the exam.
10. Create monitor blocks or auto-assign blocks.
11. Watch live monitoring during active exam.
12. Review results, reports, and violations.

### Admin Feature Areas

- Dashboard
- Notifications
- Profile/Security
- Staff creation wizard
- Staff request generation
- Staff listing
- Student admissions and management
- Question bank
- Passage management
- Exam creation and publishing
- Monitoring block assignment
- Live proctoring/monitoring
- Attempt controls
- Result review
- Reports
- School settings

### Admin Permission Model

Permission is controlled in two layers:

1. hard business logic in controllers/models
2. sidebar section permission matrix from `settings` or `school_settings`

Important examples:

- `canManageExams()`
- `canMonitorExams()`
- `canAccessSidebarSection()`

### Internal Admin Sub-Role Behavior

Current behavior from the codebase:

- `school_admin`: strongest school-level role
- `sub_admin`: management role with broad operational access
- `invigilator`: focused live monitoring role
- `staff`: support role, now also eligible in the current monitoring flow

## 3. Student Flow

### Authentication Flow

Student routes live under `/student`.

Student flow includes:

- login
- dashboard
- logout
- password update

### Typical Student Journey

1. Login to student panel.
2. Open dashboard.
3. Review system check and instructions.
4. Open exam list or mock exam list.
5. Start live exam.
6. During exam, send heartbeat updates and signaling data.
7. Violations are logged if suspicious actions occur.
8. Submit exam.
9. View result and marksheet.

### Student Feature Areas

- Dashboard
- E-learning page
- Instructions page
- System check
- Live exam runner
- Mock exams
- Exam history
- Result history
- Notifications
- Profile

## 4. Approval and Request Flow

### Staff Request Flow

Current school-side staff onboarding is approval-based:

1. Admin opens staff creation wizard.
2. Wizard captures basic info, professional details, role, and password.
3. Data is saved into `staff_requests`.
4. Superadmin reviews the request.
5. On approval, the request is expected to become a real admin/staff account.

### Admin Request Flow

Admin-side request flow is used for actions like:

- block user
- unblock user

This flow records:

- requester
- target user
- reason
- approval status

## 5. Monitoring and Proctoring Flow

### Monitoring Blocks

The monitoring system currently uses `exam_monitor_blocks` and the pivot table `exam_monitor_block_student`.

Flow:

1. An exam is created.
2. Students are grouped into blocks.
3. Each block can be assigned to an assignee using polymorphic fields:
  `assignee_type` and `assignee_id`.
4. Attempts can also store `monitor_block_id`.
5. Monitoring screens show only relevant students/attempts depending on assigned scope.

### Live Monitoring

Live monitoring uses:

- `exam_attempts`
- `exam_streams`
- `exam_violations`
- monitor block mapping

### WebRTC-Style Signaling Flow

Current stream flow appears to be:

1. Viewer requests stream for an attempt.
2. Stream record is created in `exam_streams`.
3. Offer/answer and ICE candidates are exchanged through poll and signal endpoints.
4. Viewer scope is enforced using attempt/block-based access rules.

## 6. Settings Flow

There are two settings layers:

- global settings in `settings`
- school-specific settings in `school_settings`

This lets the platform support both:

- system-wide defaults
- school-level overrides
